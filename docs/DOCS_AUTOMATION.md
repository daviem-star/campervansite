# Documentation Automation

This repository already had strong manual planning docs. The gap was that the most useful orientation breadcrumbs were spread across commit history, PRs, and the codebase itself, so each new development session still had to re-analyse the tree.

The automation added here keeps that breadcrumb trail current without trying to auto-rewrite roadmap or QA judgment calls.

## What Is Generated

- `docs/PROJECT_STATUS.md`
  - Current branch snapshot
  - Product phase status pulled from `docs/PRODUCT_PLAN.md`
  - Current routes, API surface, key files, env dependencies, QA backlog counts
  - Recent substantive commit summary
- `docs/RECENT_WORK.md`
  - Commit timeline generated from `git log`
  - Repo-history active areas so GitHub shows where momentum has been

These files are generated by `scripts/docs/generate-project-status.mjs`.

## What Stays Manual

- `README.md`
  - High-level repo entrypoint and setup summary
- `docs/PRODUCT_PLAN.md`
  - Product state, milestone order, and what we should do next
- `docs/FOUNDATION_ACTIVATION.md`
  - Hosted setup and smoke-test runbook
- `docs/QA_NOTES.md`
  - Manual QA evidence, issue log, and fix verification

These docs should only be updated when the product truth changes. They are intentionally not rewritten from commit diffs.

## Local Automation

`npm install` now configures repository git hooks by setting `core.hooksPath` to `.githooks`.

The hooks do two jobs:

- `pre-commit`
  - Regenerates `docs/PROJECT_STATUS.md` and `docs/RECENT_WORK.md`
  - Stages those generated docs automatically
- `pre-push`
  - Fails the push if generated docs are stale

If you need to re-run hook setup manually:

```bash
npm run hooks:install
```

If you want to regenerate or verify docs directly:

```bash
npm run docs:generate
npm run docs:check
```

## GitHub Automation

The GitHub workflow in `.github/workflows/docs-breadcrumbs.yml` checks generated docs on every pull request and on pushes to `main`.

The PR template in `.github/pull_request_template.md` adds the one thing automation cannot infer reliably: the human breadcrumb for the next session. Each PR should leave a short note covering:

- current status after this change
- next recommended task
- known risks or open questions
- validation run

## Recommended Team Rule

Use this split:

- Generated docs explain where the codebase is.
- Manual docs explain what we believe, what we still need to do, and what has been verified.
- PR descriptions explain what the next session should pick up.

That gives you a useful GitHub readout at any time without letting automation invent product decisions on your behalf.
